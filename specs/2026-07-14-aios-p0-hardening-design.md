# AIOS P0 收口加固设计

> 版本：v1.0 | 日期：2026-07-14 | 状态：已批准执行

## 目标

在不扩展到对话、语音、长期记忆或审核中心实现的前提下，让 P0 具备安全的测试隔离、明确的 PATCH 语义、可解析的移动端壳、可复现的开发环境和可追踪的项目记忆。

## 方案比较

### 方案 A：PostgreSQL 测试库 + FastAPI 依赖覆盖（采用）

使用现有 `db_test` 容器，测试创建独立 async engine，并覆盖 `get_db`。每个测试运行在外层事务中，API 内部 `commit()` 只影响 savepoint，测试结束统一回滚。测试夹具只幂等创建 schema，不自动执行 `drop_all`；schema 的销毁只能由明确的测试库维护命令完成。优点是行为与生产 PostgreSQL/pgvector 一致，且不会污染开发库；缺点是完整测试依赖 Docker。

### 方案 B：SQLite 内存测试库

启动快，但 UUID、PostgreSQL 方言、pgvector 和事务语义均与生产环境不同。它会让测试通过却无法证明 PostgreSQL 路径可靠，因此不采用。

### 方案 C：共享开发数据库并清理数据

改动最少，但测试异常退出时可能遗留数据，且存在误删开发数据风险，因此不采用。

## 任务边界

1. 数据库测试隔离：测试必须显式使用 `database_url_test`，并对测试 DSN 做安全校验。
2. 项目更新语义：省略 `description` 表示不变，显式 `null` 表示清空；显式 `name: null` 返回 422。
3. Web API 客户端：错误对象、422 数组和非 JSON 错误都产生可读消息；生产 API 地址允许环境变量覆盖。
4. 移动端壳：删除无效资源引用，声明真实依赖，生成 lockfile，不再虚假显示已连接后端。
5. 环境与记忆：固定 Python/Node 基线，提供环境变量示例，扩充忽略规则，并在项目记忆中记录验证事实和阻塞。

## 执行隔离

当前项目尚无 Git 仓库。先检查忽略规则并建立基线提交，再执行写入任务。实现子智能体顺序运行，每个任务拥有不重叠的文件范围；每项完成后先做规格审查，再做代码质量审查。数据库、前端和移动端验证由主线程最终统一执行。

## 验收标准

- 测试依赖覆盖不再引用开发 session，测试 DSN、用户名或实际连接身份不符合测试库约束时立即拒绝运行。
- 项目描述的省略、设置和清空三态均有测试。
- Web 客户端错误格式化有单元测试，生产构建通过。
- Expo 配置可解析，TypeScript 通过；若网络依赖阻塞，必须明确记录未验证项。
- `docker compose config`、Alembic head、后端非数据库测试保持通过。
- `docs/PROJECT_MEMORY.md` 记录本轮变更、环境版本、验证结果和剩余阻塞。
